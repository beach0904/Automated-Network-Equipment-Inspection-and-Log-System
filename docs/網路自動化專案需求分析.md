# 專案需求分析與技術規格書 (PRD)

## 專案名稱：電信級跨區域網路自動化維運與大數據監測系統
**版本：** v1.3.0 (Real-World Data Integration)  
**更新日期：** 2026-05-15

---

## 1. 專案背景與動機
在電信級或跨廠區（如：高雄小港、前鎮、林園）的網路管理環境中，傳統的人工巡檢面臨以下技術瓶頸：
* **數據孤島 (Data Silos)：** 各廠區巡檢結果分散在各個日誌檔，缺乏統一的結構化存儲，難以進行長期趨勢分析。
* **數據品質不一：** 真實設備產出的數據常包含缺失值或格式混亂，人工清洗耗時費力。
* **無效告警 (Alarm Fatigue)：** 缺乏自動化過濾機制，導致管理人員難以從海量數據中快速定位真正故障。

本專案旨在透過 **Python 自動化技術** 結合 **3NF 關聯式資料庫**，打造一套具備數據清洗、持久化存儲與智慧化分析的網路監測解決方案。

---

## 2. 技術棧 (Technology Stack)
本專案採用模組化設計，確保系統穩定性與開發效率：

| 模組 | 使用技術 | 專業定位說明 |
| :--- | :--- | :--- |
| **主程式語言** | **Python 3.1x** | 核心邏輯處理、ETL 流程控管、物件導向開發 (OOP)。 |
| **數據存儲** | **SQLite3** | 實作 **3NF (第三正規化)** 關聯式架構，確保數據一致性。 |
| **數據分析** | **Pandas** | 負責大規模真實數據 (CSV/XLSX) 的清洗、轉換與統計。 |
| **報表產出** | **Openpyxl** | 自動化生成帶有條件格式化（標色渲染）的專業維運報告。 |
| **數據獲取** | **Kagglehub / Glob** | 接軌真實世界數據集，實現批次自動化檔案偵測。 |
| **安全性** | **Parameterized Query** | 採用 SQL 預留位置技術，杜絕 SQL 注入攻擊 (SQL Injection)。 |

## 3. 系統架構設計
系統分為四個核心處理階層，實現數據從「原始資料」到「管理決策」的轉化：

### A. 數據採集層 (Data Collection & Discovery)
* **機制：** 透過 `glob` 模組自動掃描 `source_data` 資料夾。
* **數據源：** 串接 Kaggle 真實 IT Performance 數據集，模擬真實 SNMP 輪詢結果。

### B. 數據解析與 ETL 層 (Processing & Cleaning)
* **清洗邏輯：** 使用 Pandas 實作 `dropna()` 與 `fillna()`，排除真實數據中的無效噪點。
* **轉換邏輯：** 將非結構化數據對齊至專案標準格式（設備名稱、管理 IP、CPU/記憶體指標）。

### C. 持久化存儲層 (3NF Storage - SQLite3)
* **Table: `Areas`** - 抽離區域維度，避免地名重複儲存造成的冗餘。
* **Table: `Devices`** - 儲存設備資產靜態資訊（IP, Area ID）。
* **Table: `PerformanceLogs`** - 儲存高頻動態指標，實作一對多關聯。

### D. 應用與展現層 (Presentation)
* **智慧診斷：** 根據動態閥值判定 `CRITICAL` (斷線) 與 `WARNING` (高負載)。
* **視覺化報表：** 使用顏色引導技術 (Color-Guided Reporting) 產出 Excel，提升維運直覺。

---

## 4. 功能清單 (Feature List)
1. **批次異質導入：** 支持同時掃描並處理多個廠區、不同格式的數據來源。
2. **自動化清洗：** 系統具備容錯機制，自動修復或跳過真實數據中的髒資料。
3. **高效 SQL 查詢：** 透過多表 JOIN 關聯，實現「全區域異常分析」的一鍵產出。
4. **全列視覺渲染：** 異常紀錄在 Excel 中自動以紅、橘兩色區分，達成「管理例外化」。

---

## 5. 未來擴展性 (Scalability)
* **告警自動化：** 規劃對接 Line Notify 或 Telegram Bot 實現即時故障推送。
* **Web 化儀表板：** 結合 Flask 或 Streamlit 建立 Web 版網管監控中心。
* **預測性維護：** 引入簡單回歸模型，預測設備未來 24 小時的資源負載趨勢。
